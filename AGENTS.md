## 代码原则

- 函数(动作)应该以动词命名, 而文件名与类名应该以动名词命名, 以明确其职责
  - eg: 如果有一个类负责发送 email, 可以命名为 EmailSender(类名) / email_sender(文件名), 其中的函数命名为 send_email
- 所有的文件/目录/函数使用小写+下划线命名; 类使用大驼峰命名; 常量全大写; 内部函数使用下划线作前缀
- 在不引起歧义的情况下命名应该尽可能简单, 但如果一个命名同时使用于不止一个函数 / 类 / 文件, 就要进一步细化命名. 例如"操作 xxx"的函数, 如果既有批量操作也有对单个对象进行的操作, 此时要用"batch/single"进行区分
- 类要么只有一个明确的功能(与类名一致), 要么是一个功能的集合; 如果是前者, 则只应该有一个与类名一致的动词作为主要函数, 并放在类开头; 如果是后者, 用 xxx_manager 命名
- 对于每个文件夹, 应该有一个同名的主要文件, 代表这个文件夹与外界的接口
- 每个文件与函数应该有注释, 说明其作用以及为什么这么设计
- 对于文件, 函数与类, 始终使用单数; 仅在对类属性进行定义时, 使用单复数来区分属性是单个对象还是对象的列表

## 目录结构

- backend: 后端所有代码
  - src: 存放所有源码
    - app: 核心 HTTP 服务
      - api: 路由层, 只负责接收请求和返回响应, 不写任何业务逻辑
      - services: 业务逻辑层, 项目的核心, 编排数据库操作和业务规则
      - models: 数据库表结构定义, 负责与数据库交互
      - schemas: 数据契约层, 定义 API 的输入输出格式与验证规则
      - core: 基础设施, 存放配置读取、数据库连接、鉴权等被所有层依赖的公共能力
        - config.py: 读取并管理所有配置
        - ...
    - workers: 后台异步任务, 存放所有不需要用户实时等待的操作(发邮件、文件处理、定时任务等)
    - shared: 被 app 和 workers 同时依赖的公共代码, 避免两者互相 import
      - models: 两个服务共用的数据库表定义
      - utils: 两个服务共用的工具函数
  - tests: 测试代码, 目录结构镜像 src/app 的结构
  - migrations: 数据库迁移历史文件, 由 Alembic 自动生成
  - scripts: 一次性脚本, 如初始化数据、数据修复等
  - pyproject.toml: 定义后端服务的核心生产依赖
  - .env.example: 配置项模板，列出所有需要的环境变量但不填真实值，提交到 Git   
  - .env: 本地真实配置，包含数据库密码、密钥等敏感信息，加入 .gitignore 不提交
  - Dockerfile: 定义如何将当前服务（前端或后端）打包成一个可运行的容器镜像，包括运行环境、依赖安装步骤和启动命令
- frontend: 前端所有代码
  - src: 存放所有源码
    - pages: 页面级组件, 与路由一一对应
    - components: 可复用的 UI 组件
    - api: 封装所有对后端的 HTTP 请求
    - store: 全局状态管理
    - utils: 通用工具函数
  - public: 静态资源(图标、字体等)
  - package.json
  - Dockerfile: 定义如何将当前服务（前端或后端）打包成一个可运行的容器镜像，包括运行环境、依赖安装步骤和启动命令
- docs: 项目文档, 包括 API 文档和架构说明           
- .gitignore
- pyproject.toml: 作为代码规范和开发工具链的统一入口, 包含全局的代码格式化规则、类型检查和代码提交规范。
- docker-compose.yml: 统一编排前端、后端、数据库、消息队列等所有服务
- ...
